# Отчет по курсовой работе: Custom Thread Pool

## Архитектура и принцип действия
В рамках курсовой работы был реализован собственный пул потоков `CustomThreadPool`, реализующий интерфейс `CustomExecutor`. 

### Механизм распределения задач (Least Loaded)
В отличие от стандартного `ThreadPoolExecutor`, который использует одну общую очередь для всех потоков, данный пул использует **массив очередей** (по умолчанию их количество равно `corePoolSize`). 

Распределение поступающих задач осуществляется по алгоритму **Least Loaded** (наименьшая загруженность). 
*Алгоритм работы продюсера:*
1. При поступлении новой задачи в метод `execute`, поток-продюсер последовательно опрашивает размер каждой `BlockingQueue` из массива (за счет внутренних счетчиков очередей `queue.size()`).
2. Задача помещается (через `offer()`) в ту очередь, размер которой на данный момент минимален.
3. Если эта очередь переполнена, срабатывает механизм отказа либо создается новый поток сверх `corePoolSize`.

Такой подход решает проблему узкого горлышка (lock contention), возникающую при конкурентном доступе множества потоков к одной очереди. Каждый рабочий поток (Worker) берет задачи из своей закрепленной очереди, что минимизирует блокировки на чтение.

### Специфика параметра `minSpareThreads`
Параметр `minSpareThreads` гарантирует наличие пула "спящих" потоков, готовых мгновенно подхватить всплеск нагрузки.
- **Алгоритм детекции:** Внутри метода `execute` до постановки задачи вычисляется количество свободных потоков: `idle = workers.size() - activeThreads.get()`. Если `idle < minSpareThreads` и общий размер не превышает `maxPoolSize`, пул **проактивно** создает новый `Worker`.
- **Проблема осцилляции (Oscillation):** При плавающей нагрузке около порогового значения пул мог бы бесконечно создавать и убивать потоки (по `keepAliveTime`). Чтобы избежать агрессивной осцилляции, новые потоки создаются только в момент поступления новых задач (`execute`), а не фоновым потоком-наблюдателем, и удаляются только после строгого простоя длительностью `keepAliveTime`.

### Механизм отказа (RejectedExecutionHandler)
Для обработки переполнения реализован интерфейс `RejectedExecutionHandler` с использованием **CallerRunsPolicy**. Обеспечивает механизм "обратного давления" (backpressure): если пул не справляется, вызывающий поток сам выполняет задачу, что естественно замедляет прием новых задач (throttling), однако может повысить p99 latency на клиенте.

## Эмпирический анализ производительности
Для оценки эффективности реализованного пула потоков был разработан класс `Benchmark.java`.

### Методология и стенд
- **Окружение (Стенд):** Процессор Intel(R) Core(TM) i3-7100U (2 физических ядра, 4 логических потока), 4GB RAM, ОС Windows 11, Java (JDK) 26.
- **Прогрев (Warmup):** Перед началом замеров выполняется холостой прогон (20 000 задач) для прогрева JIT-компилятора (перевод байткода из режима интерпретации в C2-оптимизированный машинный код).
- **Инструмент:** Использовался самописный цикл с `System.currentTimeMillis()`. *Примечание: в высоконагруженных (High-Load) Production-системах для снятия метрик следовало бы использовать JMH (Java Microbenchmark Harness) во избежание искажений из-за Dead Code Elimination и эффектов кэширования.*

*Условия тестирования: Параллельная диспетчеризация 100 000 вычислительных задач из 4 потоков-поставщиков (Producer threads).*

| Метрика (100k задач)      | java.util.concurrent.ThreadPoolExecutor | CustomThreadPool (Least Loaded) |
|---------------------------|-----------------------------------------|---------------------------------|
| **Время выполнения**      | ~154 мс                                 | ~442 мс                         |

### Анализ "предельных" значений и архитектурные ограничения
Очевидно, что реализация `CustomThreadPool` уступает стандартному `ThreadPoolExecutor` по скорости выполнения примерно в 2.8 раза. Детальный анализ архитектуры выявил **критические точки роста (недостатки)**:

1. **Накладные расходы балансировщика (Least Loaded Overhead):** Для нахождения наименее загруженной очереди поток-продюсер вынужден в цикле `for` опрашивать размер *каждой* очереди в массиве (имеющей `ReentrantLock` внутри `size()`). Это создает линейную сложность `O(N)` при вставке задачи.
2. **Избыточная синхронизация (Synchronization Overhead):** При маршрутизации каждой задачи метод `ensureCoreAndSpareThreads()` обращается к критической секции `synchronized (workers)`. Эта глобальная блокировка жестко сериализует потоки-поставщики, полностью нивелируя потенциальный выигрыш от партиционирования очередей. Стандартный `ThreadPoolExecutor` избегает этого с помощью атомарных CAS-операций (над `ctl`).

**Анализ параметров по Брайану Гетцу (Brian Goetz):**
При тестировании выявлены два абсолютно разных профиля оптимальной конфигурации (`corePoolSize` и `maxPoolSize`):
1. **CPU-bound задачи (Вычисления):** Оптимальное число потоков рассчитывается по формуле $N_{\text{threads}} = N_{\text{cores}}$. Превышение этого лимита деградирует throughput из-за накладных расходов на переключение контекста (Context Switch).
2. **IO-bound задачи (Сеть, БД, Thread.sleep):** Для блокирующего ввода-вывода оптимальный размер пула высчитывается по формуле Гетца: 
$$N_{\text{threads}} = N_{\text{cores}} \times (1 + \frac{WaitTime}{ComputeTime})$$
При симуляции через `Thread.sleep` производительность росла линейно при увеличении пула до 50-100 потоков, так как процессор не должен простаивать, ожидая завершения блокирующих операций.
