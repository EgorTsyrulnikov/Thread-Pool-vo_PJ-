# Отчет по курсовой работе: Custom Thread Pool

## Архитектура и принцип действия
В рамках курсовой работы был реализован собственный пул потоков `CustomThreadPool`, реализующий интерфейс `CustomExecutor`. 

### Механизм распределения задач (Round Robin)
В отличие от стандартного `ThreadPoolExecutor`, который использует одну общую очередь для всех потоков, данный пул использует **массив очередей** (по умолчанию их количество равно `corePoolSize`). 
Распределение поступающих задач осуществляется по алгоритму **Round Robin** (круговое обслуживание): каждая новая задача пытается быть добавленной в следующую по индексу очередь. Если выбранная очередь переполнена, пул пробует следующую. 
Это решает проблему узкого горлышка (lock contention), возникающую в высоконагруженных системах при конкурентном доступе множества потоков к одной очереди. Каждый рабочий поток (Worker) закреплен за своей очередью (или случайной при создании сверх `corePoolSize`), что минимизирует блокировки.

### Механизм отказа (RejectedExecutionHandler)
Для обработки переполнения реализован паттерн Strategy через интерфейс `RejectedExecutionHandler`. В качестве варианта по умолчанию выбрана **CallerRunsPolicy** (выполнение в потоке, вызвавшем `execute`). 
**Почему именно этот подход:**
- **Плюсы:** Это обеспечивает механизм "обратного давления" (backpressure). Если пул потоков не справляется с нагрузкой, вызывающий поток (например, поток обработки HTTP-запросов) сам начинает выполнять задачу, тем самым естественным образом замедляя прием новых задач.
- **Недостатки:** Замедление вызывающего потока может привести к таймаутам на уровне приема запросов клиента.

## Анализ производительности
Было проведено сравнение `CustomThreadPool` со стандартным `java.util.concurrent.ThreadPoolExecutor`.

1. **Многопоточная вставка (submit throughput)**: Благодаря массиву очередей, `CustomThreadPool` показывает меньшую задержку при одновременной постановке задач из множества разных потоков. В стандартном пуле все потоки-поставщики блокируются на одной `BlockingQueue`.
2. **Скорость обработки (consume throughput)**: Так как каждый `Worker` читает из своей очереди, отсутствует конкуренция за лок головы очереди (head lock). Это делает нашу реализацию похожей на архитектуру ForkJoinPool (с Work-Stealing, хотя здесь воровство не реализовано). 

Аналогичные подходы (несколько очередей для снижения contention) часто применяются во внутренних компонентах серверов вроде Tomcat и Jetty при обработке NIO-селекторов.

## Исследование параметров пула
Для достижения максимальной производительности были протестированы различные конфигурации параметров:

1. **`corePoolSize`**: Оптимальное значение зависит от характера задач. Для CPU-bound задач идеально `corePoolSize = Runtime.getRuntime().availableProcessors()`. Для IO-bound задач (БД, сеть) размер должен быть значительно больше, чтобы компенсировать простои на ожидании ввода-вывода.
2. **`queueSize`**: Слишком малый размер очереди приводит к частому отказу (rejection) или созданию новых потоков, что увеличивает накладные расходы на context switch. Слишком большой — увеличивает задержку (latency) и потребление памяти. Наилучшие результаты получены при размере очереди, способной вместить задачи за 0.5-1 секунды работы пула.
3. **`minSpareThreads`**: Данный параметр позволяет системе быстрее реагировать на резкие всплески (spikes) нагрузки. Поддержание 10-20% потоков в горячем резерве значительно снижает задержку 99-го перцентиля (p99 latency) при волнообразной нагрузке, так как не тратится время на аллокацию системных ресурсов для новых `Thread`.
